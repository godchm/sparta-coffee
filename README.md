# Sparta Coffee

포인트 기반 커피 주문/결제 서비스입니다.

커피 메뉴 조회, 포인트 충전, 주문 생성/결제, 인기 데이터 조회 기능을 제공합니다.  
인기 데이터는 사용자의 검색 관심도를 보여주는 **인기 검색어**와 실제 결제 데이터를 기반으로 한 **인기 메뉴** 두 가지로 분리해 구현했습니다.
인기 검색어 구현 학습을 해보고 싶어서 2가지로 구현했습니다.

## 1. 문제 해결 전략

### 설계 의도

커피 주문 서비스에서 가장 중요한 부분은 결제 시점의 데이터 일관성입니다.  
주문 결제는 포인트 차감, 주문 상태 변경, 주문 내역 수집 이벤트 저장이 하나의 흐름으로 처리되어야 합니다.

이를 위해 다음 전략을 선택했습니다.

- 결제는 포인트로만 수행합니다.
- 포인트 차감은 낙관적 락과 재시도를 사용해 동시성 충돌을 제어합니다.
- 주문 결제는 Redis 분산락을 사용해 다중 서버 환경에서도 중복 결제를 방지합니다.
- 주문 내역 수집 이벤트는 결제 트랜잭션 안에서 DB에 먼저 저장하고, 이후 Kafka 전송을 재시도합니다.
- 인기 검색어는 Redis ZSET으로 당일 검색 횟수를 집계합니다.
- 인기 메뉴는 Redis ZSET으로 날짜별 주문 수량을 집계하고, 최근 7일 데이터를 합산해 TOP 3를 조회합니다.

### 기술적 선택 이유

| 기술 | 선택 이유 |
| --- | --- |
| Spring Boot | REST API, 트랜잭션, 보안 구성을 빠르게 구현하기 위해 사용 |
| Spring Security + JWT | 로그인 사용자 인증과 권한 검증을 stateless 방식으로 처리 |
| JPA | 주문, 메뉴, 포인트 등 도메인 데이터를 객체 중심으로 관리 |
| MySQL | 주문, 결제, 포인트 데이터를 영속적으로 저장 |
| Redis | 분산락과 인기 검색어/인기 메뉴 랭킹 집계를 위해 사용 |
| Kafka | 주문 내역을 데이터 수집 플랫폼으로 비동기 전송하기 위해 사용 |
| Spring Retry | 낙관적 락 충돌 발생 시 재시도 처리 |
| WebSocket | 추후 실시간 알림 기능 확장을 고려해 기본 설정 유지 |

## 3. API 명세

### 인증

| Method | URL | 설명 |
| --- | --- | --- |
| POST | `/api/v1/users` | 회원가입 |
| POST | `/api/v1/auth/login` | 로그인 |
| POST | `/api/v1/auth/reissue` | access token 재발급 |
| POST | `/api/v1/auth/logout` | 로그아웃 |

### 인증 토큰 관리

- Refresh Token Cookie
refresh token은 응답 body로 직접 노출하지 않고 HttpOnly Cookie로 전달하도록 설계했습니다. 현재 프로젝트는 백엔드 중심 구현이지만, 
추후 프론트엔드와 연동될 때 브라우저의 쿠키 저장 방식을 활용해 refresh token을 관리할 수 있도록 확장성을 고려했습니다.

- 로그인 성공 시 access token은 응답 body로 전달하고, refresh token은 HttpOnly Cookie로 전달합니다.
현재 프로젝트는 백엔드 API 중심으로 구현되어 있지만, 추후 프론트엔드와 연동될 경우 브라우저 쿠키를 활용해 refresh token을 관리할 수 있도록 설계했습니다.  
이를 통해 클라이언트가 refresh token을 직접 저장하거나 JavaScript로 접근하는 방식을 줄이고, access token 만료 시 refresh token 기반 재발급 흐름으로 확장할 수 있습니다.
refresh token은 서버 DB에도 저장하여 로그아웃, 토큰 재발급, 토큰 탈취 의심 상황에서 기존 token을 무효화할 수 있도록 했습니다.

### 인기 기능

본 프로젝트에서는 인기 데이터를 두 가지 관점으로 제공합니다.
또한 본 프로젝트의 인기 메뉴는 주문 건수보다 실제 판매 수량을 기준으로 집계했습니다.
예를 들어 아메리카노 2잔 주문 시 score를 2 증가시켜 실제 판매량이 높은 메뉴가 상위에 노출되도록 설계했습니다.

### 인기 검색어

사용자가 검색어를 입력하면 Redis ZSET에 검색어 점수를 증가시킵니다.

- Key: `popular:search:{yyyy-MM-dd}`
- Value: 검색어
- Score: 검색 횟수

오늘 날짜 기준으로 score가 높은 검색어 TOP 10을 조회합니다.

| Method | URL | 설명 | 인증 |
| --- | --- | --- | --- |
| POST | `/api/v1/popular-ranking` | 검색어 집계 | 로그인 |
| GET | `/api/v1/popular-ranking/today` | 오늘의 인기 검색어 TOP 10 조회 | 없음 |

### 인기 메뉴

주문 결제 성공 시 주문한 메뉴 ID를 Redis ZSET에 집계합니다.

- Key: `popular:menu:{yyyy-MM-dd}`
- Value: 메뉴 ID
- Score: 주문 수량

최근 7일간의 메뉴별 주문 수량을 합산해 인기 메뉴 TOP 3를 조회합니다.

| Method | URL | 설명 | 인증 |
| --- | --- | --- | --- |
| GET | `/api/v1/popular-ranking/menus` | 최근 7일 인기 메뉴 TOP 3 조회 | 로그인 |

## 4. 기능별 구현 전략

### 메뉴 조회

활성화된 메뉴만 조회되도록 구현했습니다.  
관리자는 메뉴 생성, 수정, 삭제를 수행할 수 있고 일반 사용자는 메뉴 조회만 가능합니다.

### 포인트 충전/사용

포인트는 `user_points`에서 사용자별 잔액으로 관리합니다.  
충전, 사용, 환불 이력은 `point_histories`에 저장해 포인트 변경 내역을 추적할 수 있도록 했습니다.

포인트 사용 시 동시에 여러 결제가 발생할 수 있으므로 낙관적 락과 재시도를 적용했습니다.

- `OptimisticLockingFailureException` 발생 시 최대 3회 재시도
- 재시도 초과 시 `CONFLICT_POINT_UPDATE` 예외 반환
- 잔액 부족 시 `INSUFFICIENT_POINT` 예외 반환

### 주문 결제

결제는 다음 흐름으로 처리합니다.

1. 주문 조회
2. 주문 소유자 또는 관리자 권한 검증
3. 주문 상태 검증
4. 포인트 차감
5. 주문 상태 `PAID` 변경
6. 주문 메뉴별 인기 메뉴 이벤트 저장
7. 결제 응답 반환

동일 주문에 대한 중복 결제를 막기 위해 Redis 분산락을 적용했습니다.  
따라서 여러 서버 인스턴스에서 같은 주문 결제 요청이 동시에 들어와도 하나의 요청만 결제를 수행할 수 있습니다.

### 주문 내역 실시간 수집

결제 성공 시 주문 메뉴 데이터를 `PendingPopularRankingEvent`로 저장합니다.  
Kafka 전송은 결제 트랜잭션과 직접 묶지 않고, 저장된 이벤트를 별도 발행 로직에서 처리합니다.

이 방식을 선택한 이유는 Kafka 장애가 발생하더라도 결제 데이터가 롤백되지 않도록 하기 위함입니다.  
결제는 핵심 트랜잭션으로 보장하고, 이벤트 전송은 재시도 가능한 후속 처리로 분리했습니다.

### 인기 검색어와 인기 메뉴 분리

인기 검색어와 인기 메뉴는 목적이 다릅니다.

- 인기 검색어: 사용자의 관심도 기반 데이터
- 인기 메뉴: 실제 결제된 주문 기반 데이터

따라서 두 데이터를 같은 랭킹으로 섞지 않고 별도 Redis ZSET key로 관리합니다.

## 5. 동시성 및 데이터 일관성

### 포인트 동시성

동시에 여러 결제가 발생하면 같은 포인트 잔액을 차감하려는 충돌이 발생할 수 있습니다.  
이를 해결하기 위해 포인트 엔티티에 낙관적 락을 적용하고, 충돌 발생 시 Spring Retry로 재시도합니다.

### 주문 중복 결제

동일 주문에 대한 중복 결제를 막기 위해 Redis 분산락을 사용했습니다.  
락 획득에 실패하면 결제 요청을 실패 처리하고, 예외 상황에서도 락이 해제되도록 구성했습니다.

### 이벤트 전송 일관성

주문 결제 성공 후에는 인기 메뉴 집계를 위해 주문 내역을 Kafka로 전송해야 합니다.  
하지만 Kafka 서버 장애, 네트워크 오류, 일시적인 broker 연결 실패가 발생할 경우 결제 트랜잭션까지 실패시키면 사용자는 정상적으로 결제할 수 없게 됩니다.

이를 방지하기 위해 결제 로직과 Kafka 전송 로직을 직접 결합하지 않았습니다.

결제 트랜잭션에서는 이벤트를 DB에 저장하고, 스케줄러가 짧은 주기로 Kafka에 발행하도록 구성했습니다.  
이를 통해 Kafka 장애 상황에서도 결제 데이터는 보존하고, 이벤트는 준실시간으로 재처리할 수 있습니다.

이후 별도의 발행 로직이 `PENDING` 상태의 이벤트를 조회해 Kafka로 전송합니다.

전송에 성공하면 이벤트 상태를 `SENT`로 변경하고, 실패하면 실패 횟수와 에러 메시지를 저장합니다.  
재시도가 필요한 이벤트는 `next_retry_at` 값을 기준으로 다시 발행 대상이 됩니다.

이 구조를 통해 다음 상황을 방어할 수 있습니다.

- Kafka 서버가 일시적으로 중단된 경우
- Kafka broker 연결이 실패한 경우
- 이벤트 전송 중 네트워크 오류가 발생한 경우
- 애플리케이션 재시작으로 인해 메모리 큐에 있던 이벤트가 사라질 수 있는 경우

이벤트를 메모리가 아니라 DB에 저장하기 때문에 애플리케이션이 재시작되어도 전송되지 않은 이벤트를 다시 조회할 수 있습니다.  
따라서 Kafka 장애가 발생해도 결제 데이터는 정상적으로 저장되고, 이벤트는 이후 재처리할 수 있습니다.

즉, 결제는 핵심 트랜잭션으로 보장하고, Kafka 전송은 실패해도 복구 가능한 후속 처리로 분리했습니다.

## 6. 에러 응답

공통 응답 형식을 사용합니다.

성공 응답 예시:

```json
{
  "success": true,
  "status": 200,
  "data": {}
}
```

실패 응답 예시:

```json
{
  "success": false,
  "code": "POINT_409_001",
  "message": "포인트 변경 중 충돌이 발생했습니다."
}
```

도메인별 예외는 `ErrorCode`로 관리합니다.

## 7. 테스트

작성한 테스트는 다음 제약사항을 검증합니다.

- 포인트 부족 시 결제 실패
- 포인트 낙관적 락 충돌 시 재시도
- 재시도 초과 시 `CONFLICT_POINT_UPDATE` 예외 발생
- Redis 분산락으로 동일 주문 중복 결제 방지
- 예외 발생 후 Redis 락 정상 해제
- Kafka 전송 실패 시 Pending 이벤트 저장
- Pending 이벤트 재처리
- Refresh Token 재발급 흐름

테스트 실행:

```bash
./gradlew test
```
